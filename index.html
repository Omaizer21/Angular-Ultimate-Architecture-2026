The Angular v20 Naming Revolution
The biggest convention shift in Angular history. File names and class names no longer use type suffixes. Know this before everything else.
⚠️ Most guides are still showing the OLD convention

.component.ts, UserProfileComponent, HeroService are pre-v20 patterns. Angular CLI v20 now generates user-profile.ts with class UserProfile by default. This guide uses the new convention throughout.

💡 Why did Angular change this?

The Angular team: "A class name should reflect its behavior and responsibilities, not how it is used." UsersApi tells you far more than UserService. IDEs are powerful enough to identify file roles without name-encoded suffixes. Opt back in via angular.json schematics if your team prefers the old style — consistency within your codebase matters more than which convention you use.

❌ Old pattern (pre-v20)
TS auth.service.ts
@Injectable({ providedIn: 'root' })
export class AuthService {
  constructor(private http: HttpClient) {}
  login(email: string, pw: string) {
    return this.http.post('/auth/login', { email, pw });
  }
}
✅ New pattern (Angular v20+)
TS auth.ts — class: Auth
@Injectable({ providedIn: 'root' })
export class Auth { // ✅ no "Service" suffix
  private readonly http = inject(HttpClient);
  login(email: string, pw: string) {
    return this.http.post('/auth/login', { email, pw });
  }
}

🔒
core/ — The Nervous System
One instance · Always alive · App-wide concern

Core is for things that need to exist exactly once for the entire app lifetime. Ask: "if I deleted every feature, would I still need this?" If yes → core.

✅ Belongs in core/
├─ auth.tslogin state, token
├─ auth.guard.tsblock unauth routes
├─ auth.interceptor.tsadd Bearer token
├─ error.interceptor.tscatch 401/500 globally
├─ seo.tsset <title> & meta
└─ analytics.tsGA / Mixpanel wrapper
❌ Does NOT belong
❌ UserCard → it's UI → shared/
❌ ProductsApi → feature-specific → features/products/
❌ Button → reusable UI → shared/
If deleting a feature makes this irrelevant, it belongs in that feature.
Why only one instance matters

Your Auth service holds the user's login token. If two instances exist, one has the token and the other is empty. Any component that injects the empty one thinks nobody is logged in. This is the classic CoreModule double-import bug — see the DI section below for the full story.

🔷
shared/ — The IKEA Catalogue
Reusable · Dumb · No domain knowledge · No business logic

Think IKEA furniture — works in any room, doesn't care which house. Ask: "can I copy this folder into a completely different Angular project and it still works with zero changes?" If yes → shared.

✅ Belongs in shared/
├─ 📁 button/used everywhere
├─ 📁 modal/generic overlay
├─ 📁 spinner/
├─ 📁 data-table/generic table
├─ 📁 empty-state/
└─ 📁 pipes/date, truncate, money
❌ Does NOT belong
❌ UserCard → knows about User type → features/users/
❌ OrderSummary → knows about orders → features/orders/
❌ Auth → singleton → core/
If it needs to know your domain model, it's not shared.
🚀
features/ — The Rooms in the House
Self-contained · Lazy-loaded · Owns its data · Never imports siblings

Each feature is a complete vertical slice: pages, child components, API service, state store, models, routes. Features never import from each other. If two features need something, it moves to shared/.

📁 users/complete vertical slice
├─ 📁 user-list/routable page
├─ 📁 user-card/used only within users
├─ users-api.tsHTTP: /api/users
├─ users.store.tsstate: list, loading, selected
├─ user.models.tsinterface User {id, name, role}
└─ users.routes.tslazy routes
💡 The cross-feature problem

If orders/ needs to show a user's name, it gets the user ID as an input() — it does NOT import from users/. If multiple features need the same component, that component moves to shared/.

🛷
layout/ — The Walls and Ceiling
Persistent chrome · Route parents · Always visible when authenticated

Your persistent navigation, header, sidebar. Used as route parents in the router config — not imported by feature components. The MainLayout wraps authenticated pages; AuthLayout wraps login/register.

├─ 📁 main-layout/sidebar + header + router-outlet
├─ 📁 auth-layout/centered card for login pages
├─ 📁 header/top bar, avatar, notifications
└─ 📁 sidebar/nav links to features
🔴 The CoreModule Double-Import Bug (NgModule era)
Classic NgModule problem — understanding it explains why modern Angular moved to standalone + providedIn:'root'
❌ What went wrong in NgModules
// AppModule — correct, import once
@NgModule({
  imports: [CoreModule] // ✅ one instance created
})
export class AppModule {}

// UsersModule — WRONG, imported again!
@NgModule({
  imports: [CoreModule] // 💥 second instance!
})
export class UsersModule {}

// Now Auth has TWO instances:
// Instance #1 (AppModule): has the token ✅
// Instance #2 (UsersModule): empty, blank 💥
// UserListComponent gets instance #2 → thinks
// nobody is logged in → silent auth bugs
✅ Modern fix: providedIn:'root' (standalone)
// auth.ts — the service itself declares scope
@Injectable({ providedIn: 'root' })
export class Auth {
  /* Angular guarantees exactly one instance */
  /* no matter how many times inject(Auth) runs */
}

// Feature component — just inject, that's it
export class UserList {
  private readonly auth = inject(Auth);
  // ✅ Always the same global instance
  // No module to import. No footgun.
}

// There is no CoreModule to double-import.
// core/ is just a folder convention now.
✅ Why the problem is completely gone in standalone Angular

providedIn:'root' registers the service directly with the root injector at class definition time. Angular's DI system deduplicate this — even if 50 components call inject(Auth), they all receive the same singleton. There is no CoreModule to accidentally import twice. core/ in a modern app is simply a folder that groups related singleton files — no NgModule, no forRoot(), no footgun.

🔒 CoreModule — what it actually was vs what it is now
📁 NgModule era — CoreModule was a real class
// core/core.module.ts — a real NgModule
@NgModule({
  // Components declared here were singletons
  // in the sense that the module loaded once
  providers: [
    Auth,          // ← listed explicitly
    Seo,
    Analytics,
    {
      provide: HTTP_INTERCEPTORS,
      useClass: AuthInterceptor,
      multi: true
    }
  ]
})
export class CoreModule {
  // forRoot() pattern to prevent double-import
  static forRoot(): ModuleWithProviders<CoreModule> {
    return {
      ngModule: CoreModule,
      providers: [Auth, Seo]
    };
  }
}

// app.module.ts — import ONLY here, once
@NgModule({
  imports: [
    CoreModule.forRoot() // ← forRoot = once only
  ]
})
export class AppModule {}

// If ANY feature module imported CoreModule
// again → duplicate service instances → bugs
📁 Standalone era — core/ is just a folder
// core/auth/auth.ts — no NgModule at all
@Injectable({ providedIn: 'root' })
export class Auth {
  // 'providedIn: root' IS the singleton guarantee
  // Angular deduplicates this automatically
  // No module. No forRoot(). No footgun.
}

// core/http/auth.interceptor.ts
export const authInterceptor: HttpInterceptorFn =
  (req, next) => { /* add token */ };

// app.config.ts — register providers once
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([authInterceptor])
    ),
    // Auth, Seo etc. use providedIn:'root'
    // so they don't even need listing here
  ]
};

// CoreModule.ts does not exist anymore.
// core/ is a naming convention only.
🔷 SharedModule — the biggest structural change
NgModule era — SharedModule had to be imported in EVERY feature module
// shared/shared.module.ts
@NgModule({
  imports: [CommonModule],
  declarations: [
    // Every shared component declared here
    ButtonComponent,
    ModalComponent,
    SpinnerComponent,
    DateFormatPipe,
  ],
  exports: [
    CommonModule,  // re-export so features
    // don't need to import CommonModule too
    ButtonComponent,
    ModalComponent,
    SpinnerComponent,
    DateFormatPipe,
  ]
})
export class SharedModule {}

// users/users.module.ts — must import SharedModule
@NgModule({
  imports: [SharedModule], // ← every. feature. module.
  declarations: [UserListComponent]
})
export class UsersModule {}

// products/products.module.ts
@NgModule({
  imports: [SharedModule], // ← again. and again.
  declarations: [ProductListComponent]
})
export class ProductsModule {}

Every feature module had to import SharedModule. Forget it once and your Button component just doesn't exist in that feature. Add a new shared component? Update SharedModule declarations AND exports. The boilerplate was enormous.

Standalone era — each component imports what it needs directly
// shared/button/button.ts
@Component({
  standalone: true, // ← this IS the change
  selector: 'app-button',
  // No NgModule. No declarations array.
  // No SharedModule.ts file.
})
export class Button {}

// shared/spinner/spinner.ts
@Component({ standalone: true, ... })
export class Spinner {}

// features/users/user-list/user-list.ts
@Component({
  standalone: true,
  imports: [
    Button,   // ← import the class directly
    Spinner,  // ← only what THIS component needs
  ]
})
export class UserList {}

// SharedModule.ts does not exist anymore.
// shared/ is a naming convention only.
// Each component is self-contained.

Each standalone component explicitly lists its own dependencies. No central registry. No boilerplate file to maintain. Tree-shaking is perfect because Angular knows exactly which components are used where. Add a new shared component → just use it. No module to update.

📋 Full Comparison Summary
ConceptNgModule Era (pre-v17)Standalone Era (v17+ / v20+)
CoreModule Real @NgModule class with providers array. Import in AppModule only via forRoot(). Importing it twice creates duplicate service instances. Does not exist. core/ is just a folder. Services use providedIn:'root'. Interceptors registered in app.config.ts. No duplication possible.
SharedModule Real @NgModule with declarations + exports. Must be imported in every feature module. Adding a component = update 2 arrays. Does not exist. shared/ is just a folder. Each component is standalone:true. Feature components import shared components directly by class reference.
Feature module UsersModule, ProductsModule etc. Each is a real @NgModule with declarations, imports of SharedModule, RouterModule.forChild(). Lazy-loaded as a module. Does not exist. A feature is now just a *.routes.ts file. Components are standalone. Lazy-loaded via loadComponent() or loadChildren() pointing to the routes array.
AppModule @NgModule bootstrapping root. Imports CoreModule, BrowserModule, RouterModule.forRoot(). The entry point. Replaced by app.config.ts (ApplicationConfig) + bootstrapApplication(App, appConfig) in main.ts. No NgModule at all.
Component declaration Must be declared in exactly one NgModule. Cannot be used without that module being imported first. standalone:true — the component is self-contained. Import it directly wherever you need it. No module required.
core/ and shared/ Both were real NgModule classes that Angular actually processed. Their existence had runtime implications. Pure folder conventions. Angular has no knowledge of them. They exist only to help developers organize files. You could rename them and nothing breaks.
✅ What stayed the same conceptually

The mental model of core/shared/features/layout is identical in both eras. Core = singletons, shared = reusables, features = vertical slices, layout = shell. The difference is purely in how Angular implements it — in the NgModule era you expressed it through module classes and imports arrays; in the standalone era you express it through folder structure, providedIn:'root', and direct class imports. Same concept, zero boilerplate.

TS auth.interceptor.ts — functional
import { HttpInterceptorFn } from '@angular/common/http';

export const authInterceptor: HttpInterceptorFn =
  (req, next) => {
    const token = inject(Auth).getToken();
    if (!token) return next(req);
    return next(req.clone({
      headers: req.headers
        .set('Authorization', `Bearer ${token}`)
    }));
  };

shared/ — Dumb, Pure, No Business Logic
Organized by concept, never by type. No components/, directives/, or pipes/ sub-folders — that's the official style guide rule.
📚 Official Style Guide Rule — What It Actually Means

"Avoid creating subdirectories based on the type of code that lives in those directories. For example, avoid creating directories like components, directives, and services." — angular.dev/style-guide

🧠 The Mental Model: Group by what changes together, not by what kind of thing it is

When you build a User List feature, it needs a component, a service, maybe a directive and a pipe. If you put each of those into separate type-based folders, changing anything about the User List feature means touching four different directories. The files that change together don't live together.

The rule is about cohesion. A folder should answer the question "what does this do?", not "what Angular artifact type is this?". Angular doesn't care what folder your files are in — but developers scrolling a file tree absolutely do.

❌ Type-based (what Angular tutorials used to teach)
📁 components/
├─ user-list.ts
├─ user-detail.ts
├─ product-list.ts
📁 services/
├─ users-api.ts
└─ products-api.ts
📁 pipes/
└─ date-format-pipe.ts

⚠ To understand the "users" feature you must jump between 3 folders. Deleting the users feature requires hunting across the whole tree.

✅ Concept-based (official style guide)
📁 users/everything users in one place
├─ 📁 user-list/
├─ 📁 user-detail/
├─ users-api.ts
📁 products/everything products in one place
├─ 📁 product-list/
└─ products-api.ts
📁 shared/

✅ The entire "users" feature is one folder. Delete users? Delete one folder. New dev reads "users"? One folder to understand.

💡 The Exception: pipes/ is still acceptable as a grouping in shared/

Pipes are a special case because they are stateless transformation utilities — pure functions with no UI and no feature-specific logic. A date-format-pipe.ts doesn't belong to any feature. Grouping pipes by type inside shared/pipes/ is pragmatic and widely accepted. The rule is really about feature code: don't scatter a feature's component, service, and directive across three separate type-named folders.

export class Login {
  private readonly fb = inject(NonNullableFormBuilder);
  private readonly auth = inject(Auth);

  // ✅ NonNullableFormBuilder: values are never null
  readonly form = this.fb.group({
    email:    ['', [Validators.required, Validators.email]],
    password: ['', Validators.required],
  });

  // ✅ Bridge form state to signals for clean templates
  protected readonly isValid = toSignal(
    this.form.statusChanges.pipe(map(s => s === 'VALID')),
    { initialValue: false }
  );

    if (this.form.invalid) return;
    this.isSubmitting.set(true);
    try {
      await firstValueFrom(
        this.auth.login(this.form.getRawValue())
      );
    } finally {
      this.isSubmitting.set(false);
    }
  }
}

import { Injectable, inject, signal, computed } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Router } from '@angular/router';
import { firstValueFrom } from 'rxjs';
import { User } from './auth.models';

@Injectable({ providedIn: 'root' })
export class Auth {
  private readonly http   = inject(HttpClient);
  private readonly router = inject(Router);

  // ✅ Private writable, public readonly computed
  private readonly _session = signal<{ user: User; token: string } | null>(null);

  readonly currentUser = computed(() => this._session()?.user ?? null);
  readonly isLoggedIn  = computed(() => this._session() !== null);
  readonly token       = computed(() => this._session()?.token ?? null);

  async login(creds: { email: string; password: string }): Promise<void> {
    const res = await firstValueFrom(
      this.http.post<{ user: User; token: string }>('/api/auth/login', creds)
    );
    this._session.set(res);
    localStorage.setItem('token', res.token);
  }

  logout(): void {
    this._session.set(null);
    localStorage.removeItem('token');
    this.router.navigate(['/auth/login']);
  }

  // Called on app init — restore session from storage
  async restoreSession(): Promise<void> {
    const token = localStorage.getItem('token');
    if (!token) return;
    try {
      const user = await firstValueFrom(
        this.http.get<User>('/api/auth/me')
      );
      this._session.set({ user, token });
    } catch {
      localStorage.removeItem('token');
    }
  }

  getToken(): string | null {
    return this.token();
  }
}

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(Auth).getToken();

import { HttpInterceptorFn, HttpErrorResponse } from '@angular/common/http';
import { inject } from '@angular/core';
import { catchError, throwError } from 'rxjs';
import { Auth } from '../auth/auth';

export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(Auth);

  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401) {
        // Session expired — log out silently
        auth.logout();
      }

      if (err.status >= 500) {
        // Could inject a ToastService here
        console.error('Server error:', err.message);
      }

      // Re-throw so components can still catch locally
      return throwError(() => err);
    })
  );
};

import { setAllEntities, addEntity, updateEntity, removeEntity } from '@ngrx/signals/entities';
import { inject } from '@angular/core';
import { computed } from '@angular/core';
import { firstValueFrom } from 'rxjs';
import { Task, TaskStatus } from './task.models';
import { TasksApi } from './tasks-api';

export const TasksStore = signalStore(
  { providedIn: 'root' },
  withEntities<Task>(),
  withState({ loading: false, error: null as string | null }),

  withComputed(({ entities }) => ({
    todoTasks:       computed(() => entities().filter(t => t.status === 'todo')),
    inProgressTasks: computed(() => entities().filter(t => t.status === 'in_progress')),
    doneTasks:       computed(() => entities().filter(t => t.status === 'done')),
    totalCount:      computed(() => entities().length),
  })),

  withMethods((store, api = inject(TasksApi)) => ({
    async loadAll(): Promise<void> {
      patchState(store, { loading: true, error: null });
      try {
        const tasks = await firstValueFrom(api.getAll());
        patchState(store, setAllEntities(tasks));
      } catch (e: any) {
        patchState(store, { error: e.message ?? 'Load failed' });
      } finally {
        patchState(store, { loading: false });
      }
    },

    async create(payload: Omit<Task, 'id'>): Promise<void> {
      const task = await firstValueFrom(api.create(payload));
      patchState(store, addEntity(task));
    },

    async updateStatus(id: string, status: TaskStatus): Promise<void> {
      patchState(store, updateEntity({ id, changes: { status } }));
      await firstValueFrom(api.update(id, { status }));
    },

    async remove(id: string): Promise<void> {
      patchState(store, removeEntity(id));
      await firstValueFrom(api.delete(id));
    },
  }))
);

@Pipe({
  name: 'dateFormat',  // used as {{ value | dateFormat:'short' }}
  standalone: true,
  pure: true,          // ✅ always pure unless you have a reason
})
export class DateFormatPipe implements PipeTransform {
  transform(value: string | Date, format: 'short' | 'long' = 'short'): string {
    const date = new Date(value);
    if (isNaN(date.getTime())) return '—';

    return format === 'short'
      ? date.toLocaleDateString('en-US', { month: 'short', day: 'numeric' })
      : date.toLocaleDateString('en-US', {
          weekday: 'long', year: 'numeric',
          month: 'long',   day: 'numeric',
        });
  }
}

import { inject } from '@angular/core';
import { CanActivateFn, Router } from '@angular/router';
import { Auth } from '../auth';

export const authGuard: CanActivateFn = () => {
  const auth   = inject(Auth);
  const router = inject(Router);
  return auth.isLoggedIn() || router.createUrlTree(['/auth/login']);
};

// Role-based guard factory
export const roleGuard = (
  requiredRole: 'admin' | 'manager'
): CanActivateFn => () => {
  const auth   = inject(Auth);
  const router = inject(Router);

  if (!auth.isLoggedIn()) {
    return router.createUrlTree(['/auth/login']);
  }

  const hasRole =
    requiredRole === 'manager'
      ? auth.isManager()
      : auth.isAdmin();

  return hasRole || router.createUrlTree(['/403']);
};

TSStore
The tasks domain store lives in the tasks library — not in any app. Both admin-portal and customer-app can use it. Facade pattern: apps only call the methods on the store, never touch the API layer directly.

  • Optimistic update + rollback: UI feels instant, corrects on API failure

import { signalStore, withEntities, withState, withComputed, withMethods, patchState } from '@ngrx/signals';
import { setAllEntities, addEntity, updateEntity, removeEntity } from '@ngrx/signals/entities';
import { inject, computed } from '@angular/core';
import { firstValueFrom } from 'rxjs';
import { Task, TaskStatus, CreateTaskPayload } from './task.models';
import { TasksApi } from './tasks-api';

// Facade: apps interact only through this store, never TasksApi directly
export const TasksStore = signalStore(
  { providedIn: 'root' },
  withEntities<Task>(),
  withState({
    loading:       false,
    error:         null as string | null,
    selectedId:    null as string | null,
  }),

  withComputed(({ entities, selectedId }) => ({
    selectedTask:    computed(() => entities().find(t => t.id === selectedId())),
    todoCount:       computed(() => entities().filter(t => t.status === 'todo').length),
    inProgressCount: computed(() => entities().filter(t => t.status === 'in_progress').length),
    doneCount:       computed(() => entities().filter(t => t.status === 'done').length),
    completionRate:  computed(() => {
      const total = entities().length;
      if (!total) return 0;
      return Math.round((entities().filter(t => t.status === 'done').length / total) * 100);
    }),
  })),

  withMethods((store, api = inject(TasksApi)) => ({
    select: (id: string) => patchState(store, { selectedId: id }),
    clearSelection: ()   => patchState(store, { selectedId: null }),

    async loadAll() {
      patchState(store, { loading: true, error: null });
      try {
        const tasks = await firstValueFrom(api.getAll());
        patchState(store, setAllEntities(tasks));
      } catch (e: any) {
        patchState(store, { error: e.message });
      } finally {
        patchState(store, { loading: false });
      }
    },

    async create(payload: CreateTaskPayload) {
      const task = await firstValueFrom(api.create(payload));
      patchState(store, addEntity(task));
      return task;
    },

    async updateStatus(id: string, status: TaskStatus) {
      // Optimistic update
      patchState(store, updateEntity({ id, changes: { status } }));
      try {
        await firstValueFrom(api.update(id, { status }));
      } catch {
        // Rollback on failure — would need to store previous state
        await this.loadAll();
      }
    },

    async remove(id: string) {
      patchState(store, removeEntity(id));
      await firstValueFrom(api.delete(id));
    },
  }))
);

Migration Guide — Old to New
Upgrade an existing codebase to v20+ conventions. Most steps have automated CLI schematics.
1
Update to Angular v20+ & run update schematics
Run ng update @angular/core @angular/cli. Angular's automated migration handles most breaking changes. Check update.angular.io for your specific version path.
2
Migrate constructor injection → inject()
Run ng generate @angular/core:inject — official automated migration that converts all constructor-injected dependencies to inject() calls.
3
Replace *ngIf / *ngFor with control flow
Run ng generate @angular/core:control-flow — official automated migration from structural directives to @if, @for, @switch. Old directives are deprecated in v17+.
4
Migrate class-based guards & interceptors to functional
Class-based CanActivate and HttpInterceptor are deprecated. Replace with CanActivateFn and HttpInterceptorFn functional forms. The CLI generates functional by default.
5
Migrate @Input / @Output to signal APIs
Run ng generate @angular/core:signal-input-migration and ng generate @angular/core:output-migration. Converts decorator-based inputs to input() signals.
6
Migrate NgModules to standalone
Run ng generate @angular/core:standalone. Removes NgModule declarations and adds standalone: true to all components, directives, and pipes.
7
Rename files/classes to drop type suffixes (team decision)
No automated migration for this — it's a team decision. Use IDE bulk rename (F2 in VS Code). If you keep old suffixes, configure angular.json schematics with "type": "component". Consistency matters more than which convention you choose.

10 Common Architecture Mistakes
The most frequent pitfalls seen in real Angular codebases — with the correct fix for each.
1
Organizing by file type instead of feature area Official style guide rule

What the style guide means: When you create components/, services/, directives/ folders, you are grouping by what Angular calls the file. But features don't respect those boundaries — a User List feature needs a component and a service and maybe a directive. Splitting them across type folders means every single feature is scattered across your entire project. The guide says: group by what the code does, not what Angular type it is.

❌ Type-based folders (the old Angular tutorial way)
📁 components/
├─ user-list.ts
├─ user-detail.ts
└─ product-list.ts
📁 services/
├─ users-api.ts
└─ products-api.ts
📁 directives/
📁 pipes/

Understanding, modifying, or deleting the "users" feature requires navigating 3+ folders. New developers have no idea what the app does from reading the tree — only what Angular types exist.

✅ Feature/concept-based folders (style guide)
📁 features/
├─ 📁 users/
├─ 📁 user-list/
├─ 📁 user-detail/
└─ users-api.ts
└─ 📁 products/
├─ 📁 product-list/
└─ products-api.ts
📁 shared/reusables with no feature owner

Every file for a feature lives in its folder. New dev opens users/ and sees everything. Deleting a feature = delete one folder. Reading the tree tells you what the app does.

💡 Apply this rule everywhere — inside features too

Don't create features/users/components/ or features/users/services/ either. Keep the feature flat or use sub-folders named after sub-concepts: user-list/, user-detail/. The rule applies at every level of the tree.

Not using OnPush change detection
❌ Missing changeDetection
@Component({
  standalone: true
  // ❌ Defaults to Default (slow)
})
✅ Always OnPush
@Component({
  standalone: true,
  changeDetection:
    ChangeDetectionStrategy.OnPush
})
4
Business logic inside components
❌ Logic in component
protected calcDiscount(price: number) {
  // 20 lines of tax + discount logic
  // ❌ Business logic in component
}
✅ Extract to injectable
private readonly pricing =
  inject(PricingRules);

protected readonly discount =
  computed(() =>
    this.pricing.calculate(this.price())
  );
5
Public on template-only members
❌ public (unnecessary exposure)
public isAdmin = computed(() =>
  this.role() === 'admin'
);
// ❌ Exposed to DI/queries unnecessarily
✅ protected (style guide rule)
protected readonly isAdmin =
  computed(() =>
    this.role() === 'admin'
  );
// ✅ Template access, private to class API
6
Missing track in @for loops
❌
@for (user of users()) {
  <!-- ❌ No track — full re-render -->
  <app-user-card [user]="user" />
}
✅
@for (user of users();
  track user.id) {
  <app-user-card [user]="user" />
}

Using ngClass / ngStyle instead of binding syntax
❌ Extra overhead

✅ Official recommendation

Complete Rules Reference
Every key rule from angular.dev/style-guide plus 2026 best practices.
File & Class Naming
  • No type suffixes in filenames: user-list.ts
  • No type suffixes in class names: UserList
  • Pipes KEEP Pipe suffix. NgModules KEEP Module
  • Pipe file separator: hyphen → date-format-pipe.ts
  • kebab-case for all files and folders
  • One concept per file
  • Avoid generic names: utils, helpers, common
Project Structure
  • All UI code in src/
  • Bootstrap in main.ts always
  • Organize by feature area, not file type
  • No type-based subdirs — avoid components/, services/, directives/. Group by feature/concept, not by what Angular calls it
  • Group .ts + .html + .scss + .spec together
  • Tests colocated, never in top-level /tests
Components
  • standalone: true always
  • OnPush change detection always
  • input() / output() not @Input/@Output
  • All Angular APIs: readonly
  • Template-only members: protected
  • Angular props grouped before methods
  • Implement lifecycle interfaces
Services / DI
  • inject() over constructor injection
  • providedIn: 'root' for singletons
  • Domain-specific names: AuthClient, UsersApi
  • One responsibility per injectable
  • Feature-scoped: provide in route
  • Never instantiate with new
Routing
  • Always lazy-load with loadComponent()
  • Functional guards, resolvers, interceptors
  • Wildcard route ** always last
  • withComponentInputBinding()
  • One *.routes.ts per feature
  • Layout components as route parents
Signals (Stable v20)
  • RxJS fetch → toSignal() to display
  • Derived state → computed()
  • Writable dependent → linkedSignal()
  • Side effects → effect() (sparingly)
  • Complex template logic → computed
  • No takeUntil with signal state
Templates
  • @if, @for, @switch not structural directives
  • track expression in every @for
  • [class.foo] not [ngClass]
  • [style.prop] not [ngStyle]
  • Name handlers for what they do: saveUser()
  • @defer for below-fold content
TypeScript
  • strict: true in tsconfig — mandatory
  • No any — use generics/types
  • Interfaces for shapes, no I prefix
  • const / readonly over mutability
  • Optional chaining ?. and nullish ??
  • Target ES2022+ in tsconfig
